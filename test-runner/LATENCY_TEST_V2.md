# Latency Test v2

## What we're measuring

Per-packet DERP relay round-trip latency at different background load levels. This is the relay's contribution to latency — no WireGuard, no Tailscale, just DERP protocol ping/echo through the relay.

## How it works

The `derp_test_client` binary has `--mode ping` and `--mode echo`. A ping client sends a DERP frame containing a timestamp to an echo peer. The echo peer immediately bounces it back. The ping client measures the round-trip time from the embedded timestamp.

The path is: ping client → relay → echo peer → relay → ping client. Two relay traversals per measurement.

```
Client-1 (ping)             Relay              Client-2 (echo)
    │ SendPacket(echo_key)    │                      │
    │────────────────────────→│ forward to echo       │
    │                         │─────────────────────→│
    │                         │       echo back       │
    │                         │←─────────────────────│
    │       RecvPacket        │                      │
    │←────────────────────────│                      │
    │ RTT = recv_ts - send_ts │                      │
```

### Client roles

| Client | Role |
|--------|------|
| Client-1 | Ping probe (sends pings, measures RTT) |
| Client-2 | Echo responder (bounces packets back) |
| Client-3 | Background load generator |
| Client-4 | Background load generator |

The ping/echo pair is on separate VMs from the load generators. The ping probe doesn't generate bulk traffic — it only sends small 64-byte ping frames.

### Echo management

The echo responder must be restarted before every ping run. Without this, the relay's routing becomes stale after the previous ping disconnects, causing "warmup echo timeout" on the next run. This was the root cause of the 50% failure rate in earlier attempts.

Pattern: kill echo → start fresh echo → capture new key → run ping.

### Load levels

Background traffic generated by clients 3-4 using `derp_scale_test` with pair files. Rates scaled to % of TS TLS ceiling at each config.

| Level | Rate |
|-------|------|
| Idle | 100 Mbps (not 0 — avoids unlimited flood) |
| 25% | 25% of TS ceiling |
| 50% | 50% of TS ceiling |
| 75% | 75% of TS ceiling |
| 100% | 100% of TS ceiling |
| 150% | 150% of TS ceiling |

TS ceiling probed at each config with 3 runs × 6 rates before the latency measurements begin.

## Analysis per data point

With 10 runs × 4500 samples = 45,000 samples per load level per server:

- **Percentiles from pooled samples**: p50, p90, p95, p99, p999, max
- **Per-run p99 with 95% CI** across the 10 runs (shows variability)
- **Per-run max** — flag if any run's max > 10× median max (hypervisor event)
- **Load-latency curve**: p50 and p99 vs load level, HD and TS on same plot

### Expected results

- HD: flat or gently rising latency up to 100% of TS ceiling (the relay is below its own ceiling). Slight rise at 150%.
- TS: steeper rise due to goroutine scheduling overhead under load. GC pauses should appear as p999 outliers.
- Both: ~80-120μs baseline (GCP VPC RTT between VMs).

### What makes this test credible

- 4500 measured samples per run (500 warmup discarded)
- 10 runs per data point — proper CIs on percentiles
- 2.16 million total samples across the full suite
- Background load is real DERP traffic (same as the throughput benchmarks), not synthetic
- Ping/echo pair on separate VMs from load generators — latency isn't self-inflicted
- Fresh echo per run — no stale routing artifacts
